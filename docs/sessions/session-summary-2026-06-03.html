Session Goals

  • Evaluate whether the selected 16x9 Figma frames accurately reflect live site image sizing and cropping behavior
  • Identify any discrepancies between Figma implementation and live site behavior
  • Update the Editorial Hero image sizing documentation to reflect findings

Investigation Completed This Session

Figma frames evaluated

Seven Editorial Hero frames ending in "16x9" were evaluated across all breakpoints: 1730px, 1184px, 992px, 991px, 600px, 599px, and 352px. Design context was extracted from each frame and compared against live site extraction at matching breakpoints.

Key findings

✅
Container dimensions confirmed matching Image container dimensions in the Figma frames match the live site measurements documented in the editorial-hero-image-brief at all breakpoints within the measurement range. The known height delta at 991px and 600px (341px live vs 352px Figma) was confirmed.
✅
Cropping behavior confirmed equivalent — with one exception The Figma frames use object-fit: cover inside an aspect-ratio wrapper. The live site uses background-size: cover on a div. Both produce identical cropping behavior at desktop and mobile breakpoints where the container is wider than the source image ratio.
✕
Figma 600px frame does not match live site behavior At 600px, the container ratio (1.52:1) is taller than the source image ratio (3:2 = 1.5:1). On the live site, background-size: cover scales the image up to fill the height, cropping the sides. In the Figma frame, the aspect-ratio wrapper constrains the image to its natural 16:9 proportions — the image cannot fill the container height, and the #81A9E5 background is exposed above and below the image. This is a Figma implementation artifact, not a representation of live site behavior.
✅
Crop axis behavior understood and documented At all desktop and mobile breakpoints the container is wider than the source image ratio — the image scales to fill width and the top/bottom crop. At 600px the container is proportionally taller than the source image ratio — the image scales to fill height and the sides crop. This is the only breakpoint where horizontal content is more aggressively cropped than vertical content.
✅
Structural difference between Figma and live site confirmed as expected The live site uses a <div> with background-image — no <img> element, no aspect-ratio wrapper. The Figma frames use <img> with object-fit: cover inside an aspect-ratio wrapper. This structural difference is expected and does not affect the accuracy of the Figma frames at breakpoints where the container is wider than the source ratio.

Files Updated This Session

editorial-hero-image-brief.html — updated to v1.1

  • Crop table updated — new Crop axis column added. Each row now states whether the breakpoint is width-constrained (image scales to fill width, top/bottom crop) or height-constrained (image scales to fill height, sides crop).
  • 600px row updated — crop axis noted as height-constrained with an explicit call-out that more horizontal content is lost at this breakpoint than any other.

Key Decisions and Findings

✅
Figma frames are an accurate representation of live site behavior — except at 600px The statement "the Figma frames visually match what the live site would do with the same image and the same focal point" holds at all breakpoints except 600px, where the aspect-ratio wrapper in Figma prevents the fill-height scaling that the live site performs.
✅
Crop axis column added to documentation for Claude's benefit The crop axis (width-constrained vs height-constrained) was previously derivable only by comparing container ratio to source image ratio. It is now stated directly in the crop table, making breakpoint-specific extraction and safe zone decisions faster and more reliable.
🔲
600px Figma frame — not updated this session The 600px frame's aspect-ratio wrapper causes it to misrepresent live site behavior. Whether to update the Figma frame to reflect the correct fill-height behavior was not addressed this session.
