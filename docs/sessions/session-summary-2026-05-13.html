This is a methodology and process session summary — it records what was changed, tested, and decided during a development session. It is not an audit handoff document. Session Goals

  • Review an independent audit run by Charles and capture a finding from it
  • Discuss broader integration and scaling questions — agentic patterns, tool fragmentation, deploying the audit system to other designers
  • Address the selector registry maintenance problem and define a more sustainable approach
  • Update both skills to reflect the new approach

Files Updated This Session

kaplan-selector-registry SKILL.md — updated

  • Journey Module C border-radius note corrected — changed from "known fail" to pseudo-element delta requiring verification. The border-top-left-radius: 60px is implemented on the ::before pseudo-element, not the element directly. Query with getComputedStyle(element, '::before') to confirm.
  • Opening section reframed — registry is now explicitly a reference cache, not a required source of truth. "Use confirmed entries when available. Derive confidently when not."
  • Growth pattern documented — registry grows naturally from non-obvious derivations surfaced during audits, not from systematic pre-population.
  • Adding New Entries section rewritten — hard-won knowledge framing applied. Clear criteria for what gets registered (pseudo-element discoveries, duplicate element resolutions, unfamiliar MUI nesting) versus what does not (routine selectors re-derivable from standard MUI patterns).

kaplan-audit-extraction SKILL.md — updated

  • New section added — MUI Selector Derivation Protocol. Five-step reasoning sequence for deriving selectors when a registry entry does not exist: (1) identify MUI component type, (2) apply known nesting patterns, (3) check pseudo-elements for visual styles, (4) use partial class matching on semantic suffixes, (5) state confidence level.
  • Pseudo-element check elevated to a required step for visual style properties — not an optional fallback. Specifically: if a visual property reads unexpectedly as 0px or none on the direct element, always check ::before and ::after before classifying as a fail.
  • Confidence level classification added — "Confident" versus "Spot-check recommended." Derived selectors must be flagged with confidence level; spot-check cases must be explicit in the extraction output.
  • End of Session skill update check rewritten — shifts from "flag everything new" to "flag hard-won knowledge only." Routine selectors are not registered; real discoveries are.

Key Decisions and Findings

✅
Border-radius pseudo-element finding captured — in an independent audit of Journey Module C, the border-top-left-radius: 60px on Content Grid at tablet was initially classified as a fail because the direct element read 0px. DevTools inspection confirmed the radius is implemented on the ::before pseudo-element. This is likely a delta, not a fail. Corrected in the registry and elevated to a required extraction step in the derivation protocol.
✅
Selector registry model redefined — pre-building a comprehensive registry is not sustainable as the module library expands. New model: Claude derives selectors from first principles using a documented protocol; the registry is a reference cache for non-obvious derivations only. No maintenance burden; no gaps that block an audit.
✅
MUI Selector Derivation Protocol defined and added to extraction skill — five-step reasoning sequence covers component type identification, nesting patterns, pseudo-element checks, partial class matching, and confidence classification. This is the mechanism that makes the registry optional rather than load-bearing.
✅
Broader scaling discussion — key insight — Claude is already functioning as the central layer for the audit work (referencing Figma, extracting live site values, outputting notation back into Figma). The scaling question is not about adding new tools but about making the existing system deployable to other designers. The derivation protocol is a direct step toward that goal — designers do not need to understand MUI internals, only validate Claude's confidence-flagged output.
🔲
Forms audit — to be scoped next session — Charles is beginning a forms audit. Button states are a significant part of the scope. Forms introduce interactive states (rest, focus, filled, error, disabled), visual style properties (color, border), and animated label behavior — all of which expand beyond current methodology scope. Requires a scoping conversation before extraction approach can be designed.
