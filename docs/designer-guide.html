How to start, run, and end an audit session

This guide covers the practicalities of running an audit session — how to set up, what to do at each stage, how to hand off between conversations, and what to do when something goes wrong. For the full audit process, see the UX Audit Methodology doc. For Claude's tooling instructions, see the Audit Session Context doc.

The three session types

Before starting, know which kind of session you're running. Claude behaves differently for each — say which one at the start.

Audit Verifying a module against its Figma spec. Kicked off by pasting a Module Audit Brief.
Exploratory Investigating a Figma component or working out a new approach before it becomes a repeatable audit. Say you're exploring.
Documentation Working on the audit docs or Figma documentation. Say you're doing documentation.

The six phases

An audit runs in six phases — region mapping, Figma extraction, live site extraction, visual inspection, comparison, and annotation. Claude leads each phase; you review and confirm at each gate before it proceeds.

Phase 1 Establish the visual region map
Phase 2 Pull Figma values
Phase 3 Extract live site values
Phase 4 Visual inspection
Phase 5 Compare and document discrepancies
Phase 6 Annotation reference

Before You Start

1
Open a chat in the Claude Project — not a standalone Claude chat. The Project carries the methodology docs so Claude knows what to do. A standalone chat will not have the right context.
2
Open Figma desktop with the Figma extension active in Claude. The extension must be running — Claude uses it to read the Figma file directly.
3
Open Chrome with the Claude in Chrome extension active. Claude uses this to inspect and interact with the live site. Close DevTools before starting — having DevTools open can interfere with Claude's browser interactions.
4
Fill out the Module Audit Brief — module name, Figma frame name, live site URL, breakpoints, and any notes about known complexity. Use the Copy button to copy it to your clipboard. Open the brief form →
5
Paste the brief into the chat to kick off Phase 1. Claude will take it from there.

Resuming an audit mid-way? — paste your audit handoff block first, before anything else. See Starting a new chat / resuming below.

Running a Session

Claude leads the audit phase by phase. Your job is to review what Claude produces at each gate and confirm before it proceeds. You do not need to use browser DevTools or the terminal — everything runs in the chat.

At each phase gate

Review, then confirm explicitly

Claude will present its output at the end of each phase and wait. Read it, check it, and tell Claude to proceed when you're ready. If something looks wrong, say so — Claude will correct and re-present before moving on.

The most important gate is after Phase 1. Claude will inject colored highlights onto the live page so you can visually confirm the section boundaries are correct. Claude will then ask two questions in sequence — first whether you want to take a screenshot of the highlighted page before the highlights are removed, then whether you are ready to proceed to Phase 2. These are separate prompts; answer each one before moving on.

If you lose track

Ask Claude where you are

At any point you can ask: "What phase are we on and what's been confirmed?" Claude will summarise the current state before continuing.

Ending a Session

When you're done for the day, let Claude know — something like "I'm done for now" or "let's stop here." Claude will take it from there.

What happens at wrap-up

When you signal you're finished, Claude will:

  • Confirm what phase the audit is at and what has been completed
  • Print an audit handoff block in the chat — a short plain-text summary of the audit state
  • Ask if you want it saved as a doc

Copy the audit handoff block and keep it somewhere you can find it — a notes app, a doc, wherever works for you. You will paste it at the start of the next conversation to resume the audit. Without it, Claude will not know where you left off.

⚡ Ambiguous closings

If it is not clear whether you are wrapping up or just taking a break, Claude will ask. If you mean to finish for the day, say so explicitly so the handoff block gets generated before you close the chat.

Starting a New Chat / Resuming

The audit handoff block is for Claude, not for you

The handoff block Claude prints at the end of a conversation is not a record for you to read — it is a plain-text state snapshot for Claude to read at the start of the next conversation. It tells Claude which module, which phase, which breakpoints are done, and what the section map looks like. Without it, Claude has the methodology docs but not your audit state.

Keep it somewhere accessible — a notes app, a doc, wherever works. You will paste it at the start of the next conversation.

1
Open a new chat in the Claude Project — same Project, new conversation.
2
Paste the audit handoff block — the plain-text block from the end of your last conversation. Do this before anything else. Claude will confirm the restored state before proceeding.
3
Confirm and continue — Claude will tell you where you are and what comes next. Confirm, and the audit resumes from where you left off.

When to Start a New Chat

Start a new chat in the Project when any of the following apply:

↩
The current session has gotten long and Claude's responses feel less precise or less consistent — this is a sign the context window is filling up
↩
You are switching to a different module — each module gets its own session thread
↩
Something has gone wrong and you want a clean slate — make sure to wrap up properly first so your audit handoff block is generated before you close the chat

When Something Goes Wrong

🔍
Claude loses track of the section map — paste the audit handoff block again. Claude will re-read it and reorient.
🔍
Highlight injection fails — ask Claude to retry the injection script. If it continues to fail, confirm the Chrome extension is active and the correct tab is open.
🔍
Figma connection drops — ask Claude to re-confirm the MCP connection. Make sure the Figma desktop app is open and the extension is active.
🔍
Chrome extraction returns unexpected values — ask Claude to verify the selector and re-run. Claude will check whether the target element is the visible one at the current breakpoint.
🔍
Viewport width doesn't match the target breakpoint — Claude will re-navigate to the page and retry the resize automatically. You do not need to manually resize the window. If the issue persists after a retry, Claude will flag it.
🔍
You're not sure where you are — ask: "What phase are we on and what's been confirmed?" Claude will summarise before continuing.
