System Overview

A conceptual map of the audit system — what lives where, and how it changes

This document is a one-page mental model of the audit system. It is not a process doc — for the audit process see the UX Audit Methodology; for how to run a session see the Designer Guide. The purpose here is to make visible the four layers the system actually runs on, what each layer holds, who maintains it, and how new findings flow between them.

The four layers

What each layer does

Methodology layer

The conceptual frame — phases, scope, structural divergence taxonomy, section naming convention, the rules of how an audit runs. Lives in ux_audit_methodology.html, designer-guide.html, and session-context.html. This layer changes rarely. When it does, the change is fundamental — a new phase, a new core principle, a redefinition of what the audit covers.

Operational layer

The working rule set — confirmed selectors per module, MUI extraction patterns, mandatory verification checkpoints, derivation protocols, viewport sequencing. Lives in the two skills: kaplan-audit-extraction and kaplan-selector-registry. This layer changes often, in small increments. A new checkpoint after a Phase 5 miss, a new selector entry after a non-obvious derivation, a refinement to an extraction function — each is a small update that compounds over time.

Audit sessions

Where the methodology and the rule set get used. Designers run audits as conversations in the Claude Project; state travels between conversations via the plain-text audit handoff block. The session layer consumes the two layers above and produces findings — passes, fails, deltas — for the module being audited.

Maintenance loop

How the system gets better. Findings from sessions — misclassifications surfaced, new selectors derived, DOM exploration calls that went wrong — flow back up into the operational layer as proposed updates. Claude drafts the updates; you approve, edit, or discard them. This is the mechanism that converts individual session experience into durable rules without requiring designers to maintain documentation themselves.

Who maintains what

Designers using the system only touch the audit session layer. They run audits, generate findings, hand off state between conversations. They do not edit skills, methodology docs, or the registry.

You — as system architect, with Claude — maintain the methodology and operational layers. The maintenance loop is the rhythm: session findings surface candidate updates, you and Claude work through them at the end of a session or in a dedicated maintenance pass, and approved changes land in the relevant skill or doc.

The split matters because it determines whether the system scales. If designers had to maintain the rule set, every new designer would be a maintenance burden. If the rule set is maintained centrally and consumed downstream, every new designer is pure throughput.

How findings become rules

The mechanism for the maintenance loop is the proposal-and-approval pattern. At the end of an audit session, Claude surfaces any moments that look like candidates for a new rule, checkpoint, or registry entry — a misclassification the designer corrected, a selector that needed unusual derivation, a DOM depth call that went wrong. Claude drafts the proposed update in the format the relevant skill or doc expects. You approve, edit, or discard.

This keeps the maintenance cost on the system architect, not the designers. They run audits; corrections they generate flow back through their session summaries or audit handoffs, which Claude reads and proposes updates from when you run a maintenance session.

Two principles hold:

  • Claude proposes updates; humans approve them. Autonomous documentation drift is a failure mode — one designer's edge case becoming a rule that breaks the next three audits. The review gate is load-bearing.
  • The two upper layers update at different rhythms. The operational layer churns; the methodology layer holds steady. Keeping these separate means the operational layer can move quickly without destabilising the conceptual frame that designers depend on.

Why this matters for the agentic phase

The agentic phase — Claude running more of the audit autonomously — depends on this layered structure being healthy. Autonomy at Phase 5 (delta versus fail classification) becomes safe only when the operational layer has enough checkpoints to catch the failure modes that currently require a designer in the loop. Autonomy at Phase 3 (extraction) becomes safe only when DOM exploration depth is encoded as explicit rules rather than implicit judgment.

In other words: the path to autonomy is not a separate build. It runs through the maintenance loop. Every miss that becomes a checkpoint, every non-obvious derivation that becomes a registry entry, is a step toward making the human gate optional rather than required.
