UX Audit Methodology

How we verify design-to-development fidelity across the Kaplan component library

This document defines the principles, phases, and scope of the Kaplan UX audit system — why it is structured the way it is and what each phase is designed to achieve. For step-by-step instructions on running a session, see the Designer Guide →

This process moves design auditing from ad-hoc, eyeball-based checking to a repeatable, documented system. It creates a clear record of what was tested, what matched, and what didn't — which builds trust with development teams and makes design system conversations more concrete. Because parameters, selectors, and breakpoints are defined once and reused, each successive audit gets faster and more consistent. Over time, this creates a compounding return: less back-and-forth, fewer implementation errors, and a shared vocabulary between design and development that reduces misinterpretation at the source.

The six phases

Phase 2 Pull Figma values
Phase 3 Extract live site values
Phase 4 Visual inspection
Phase 5 Compare and document discrepancies

Audit Scope

This methodology applies to any combination of the following parameter categories. Each audit defines its scope upfront — not every category applies to every module or component.

Current scope

Typography

  • Font family
  • Font size
  • Font weight
  • Line height
  • Letter spacing
  • Text transform

Spacing

  • Padding
  • Margin
  • Gap (column and row)

Future scope

Dimensions

  • Min/max width
  • Min/max height

Visual Style

  • Background color
  • Text color
  • Border
  • Border radius

Interactive States

  • Hover
  • Focus
  • Active / pressed
  • Disabled

Layout Behavior

  • Display property
  • Alignment
  • Text wrap behavior

Extraction patterns and tooling for future scope categories are not yet fully defined. The framework is designed to accommodate them as the methodology matures.

Structural divergence and implementation deltas

Figma and the live site frequently diverge in structure while converging visually — the same rendered result achieved through different organisation. This is expected and is not itself a problem. Understanding the common patterns of divergence is essential for accurate Phase 5 classification.

Five categories of structural divergence appear across Kaplan's component library. All produce the same visual result; none require a fix.

Spacing property substitution

The same visual gap is produced by a different CSS property or on a different element. For example: Figma specifies padding-left: 40px on a section container; the live site achieves the same spacing via padding-left: 16px on the outer container and padding-left: 24px on an inner wrapper — 40px in aggregate.

Layout mechanism substitution

The same visual layout is produced by a different layout model. For example: Figma uses auto-layout with distributed spacing; the live site uses flexbox with justify-content: space-between.

Structural nesting mismatch

A Figma group boundary has no DOM equivalent, or vice versa. For example: Figma groups logo and tagline as one layer; the DOM renders them as two sibling elements with no shared parent. Or the DOM introduces a wrapper element carrying padding that has no counterpart in the Figma layer structure.

Constraint or alignment mechanism

Figma uses design constraints to express alignment; the live site uses a different CSS mechanism for the same result. For example: a Figma "stretch to fill" constraint maps to width: 100% on a child, or a Figma center alignment maps to margin: 0 auto.

Value expression difference

The same intended value is expressed differently in Figma versus CSS. For example: Figma specifies line-height: 24px; the live site uses line-height: 1.5, which resolves to 24px at the current font size. Or Figma specifies letter-spacing: 0.5px; the live site uses letter-spacing: 0.03em, which is equivalent at the current font size.

Phase 1 structure notes are the early signal for these patterns. When a divergence is flagged in the section map, it primes Phase 5 to look for an implementation delta rather than treating the value difference as a straightforward discrepancy.

Section naming convention

Section names must be consistent across all phases of an audit — Phase 1 section map, Phase 5 comparison tables, and Phase 6 annotation references must all use the same label for the same region.

Use Figma layer names where possible

When a Figma layer name clearly describes the visual region, use it as the section name. This creates a direct connection between the audit output and the Figma file, making handoff conversations more precise. If a Figma layer name is ambiguous or overly technical, agree on a plain-language label with the designer before Phase 1 is confirmed — and record it in the session summary for reuse in future sessions on the same module.

Audit Structure

Phase 1

Establish the visual region map

Before any values are measured, identify the module's visual regions — the distinct areas a designer would point to on screen and name. Regions are defined by what they do visually, not by how Figma or the DOM happens to organise them internally.

Claude identifies regions by reviewing the Figma frame and the live page together, then records how each tool represents the same region. This distinction matters: the Figma layer and the CSS selector are evidence of a region's boundary, not definitions of it. The two will not always align structurally, and that's expected.

The section map table captures all of this:

  • Section name — the visual region label, agreed between designer and developer. Use Figma layer names where possible. Must be consistent across all subsequent phases.
  • Description — plain language: what does this region contain or do?
  • Figma layer — the layer or group in Figma that represents this region
  • CSS selector — the live DOM element Claude will target for extraction
  • Structure note — flag any divergence between how Figma and the DOM organise this region (e.g. "Figma groups logo and tagline as one layer; DOM separates into two sibling elements"). Reference the relevant divergence category from the Structural divergence taxonomy above.

The Structure note column is not optional when divergence exists. It makes the mismatch visible rather than silently resolving it — and it provides the early signal that Phase 5 will use to classify a value difference as an implementation delta rather than a genuine discrepancy.

Immediately after presenting the table, Claude injects a highlight script into the live site via Chrome MCP. Each mapped section is outlined in a distinct color with a label directly on the page — no DevTools required. The highlights serve one specific purpose: confirming that the section map boundaries are correct before committing to them. Every subsequent phase depends on this mapping being right; the highlights are the only human-readable verification that it is. Claude does not attempt to capture screenshots at this or any other stage; visual review is always done by the designer directly in the browser.

After injection, Claude will ask whether you want to take a screenshot of the highlighted page before the highlights are removed. Take one if you want a record — this is the annotated view of the section boundaries and is useful for documentation and developer handoff. Once you are done, confirm to proceed. Claude will then ask for explicit confirmation to move to Phase 2.

Correct any incorrect mappings before confirming. This section map becomes the shared reference for the rest of the audit — all subsequent phases reference it by section name.

Phase 2

Pull Figma values

Using Figma MCP, extract values for every section identified in Phase 1. Use get_screenshot first to orient, then get_design_context for precise values. Record Figma values before touching the live site — Figma is always the source of truth for what is intended.

Key properties to capture per section at every defined breakpoint:

  • Spacing: padding, margin, gap (column and row)
  • Typography: font family, size, weight, line height, letter spacing, text transform

Where Figma distributes spacing across multiple frames or containers within a region, record each value individually and note the aggregate. This provides the Figma side of any implementation delta comparison in Phase 5.

Flag any properties with no Figma value before proceeding — these are gaps that need to be resolved before the audit can be considered complete.

Figma frames frequently contain placeholder text, dummy images, or color schemes that differ from the live site. Note these observations inline — e.g. "Hero section uses placeholder image" or "Background color in Figma differs from live" — but do not treat them as blockers. Continue extraction and briefly summarise any such observations at the end of Phase 2, offering to explore further if the designer wants to.

For audits that include future scope parameters, extraction patterns will be defined on a per-audit basis before Phase 2 begins.

Phase 3

Extract live site values

Using Chrome MCP, run a single comprehensive JavaScript extraction per breakpoint range rather than multiple targeted queries. The extraction should:

  • Walk every element within the component
  • Report all non-zero spacing values
  • Capture typography on representative text elements in each section

For each section, capture spacing values not only on the directly targeted element but on relevant ancestors and descendants within the region boundary. Spacing may be distributed across multiple DOM elements in the hierarchy — capturing the full hierarchy ensures Phase 5 has the data needed to compute aggregate values and make accurate implementation delta calls.

Within each breakpoint range, apply the following consistency assumptions as defaults — but treat them as defaults, not guarantees. No rule is without exceptions, and the audit brief Notes field is the place for a designer to flag known departures before the session begins.

  • Typography — assume consistent across the full range. Extract once at a representative width; do not re-extract at both ends unless flagged.
  • Vertical spacing — row gaps, padding top/bottom, and component-to-component spacing — assume consistent across the range. Extract once.
  • Internal column gaps — assume consistent across the range. Extract once.
  • Horizontal edge spacing — left and right padding on outer containers — spot-check at both ends of the range. Fluid layout behavior can cause visible edge spacing to shift continuously across a range even when no breakpoint fires. If the brief flags a section as likely fluid, extract at both ends; otherwise extract at the midpoint and note the assumption.

Desktop extraction must also account for the legacy breakpoint split — 992px–1729px and 1730px are separate ranges and must always be extracted independently. Do not assume continuity across this boundary for any property category. Cross-reference the element list against the section map to verify nothing has been missed. If an element appears in the DOM extraction but not in the section map, investigate before proceeding.

Phase 4

Visual inspection

Review the rendered module at each breakpoint before comparing values. Document anomalies that systematic extraction alone wouldn't surface — unexpected gaps, misalignment, or elements that feel spatially off. These observations feed directly into the annotation layer in Phase 6.

This step requires human judgment and cannot be automated. Claude does not capture screenshots — the designer reviews the live page directly in the browser and reports observations into the chat.

Phase 5

Compare and document discrepancies

Compare Figma values against live values section by section. Lead with a conclusion block before the full tables — a short, scannable summary of what passed, what failed, and what was classified as an implementation delta. The full tables follow as the corroborating record.

When a value difference is found, classify it before recording it:

  • Discrepancy — the aggregate visual result differs between Figma and the live site. Requires attention.
  • Implementation delta — the intended spacing value is correct in aggregate, but distributed differently across the hierarchy on one or both sides. The visual result is the same; the structural path to that result differs. Does not require a fix.

To determine whether a value difference is an implementation delta, sum spacing values across the relevant hierarchy on each side and compare the totals — do not compare individual property values in isolation. An implementation delta may exist when: the DOM distributes across multiple elements what Figma expresses as a single value; Figma distributes across multiple frames or containers what the DOM expresses as a single value; or both sides distribute differently but sum to the same total. Use Phase 1 structure notes as the primary signal — a flagged divergence pattern should prompt this additive check before classifying as a discrepancy.

The conclusion block leads the output and contains:

  • A count of passed, failed, and implementation delta findings
  • A one-line summary per implementation delta — what differs, why it is considered equivalent
  • A one-line summary per genuine discrepancy — what differs and in which section

The full comparison tables follow the conclusion block and are the canonical record. Output is organised into two independent top-level sections — Typography and Spacing — each structured the same way:

  • One subsection per logical region (matching the Phase 1 section map)
  • Within each region, one table per breakpoint range
  • Each table shows all properties — Figma value, live value, and status (pass / fail / delta) — regardless of whether they match

After presenting the full tables, Claude will ask whether you want a filtered view showing only failures and deltas — useful for focused review. Group findings by breakpoint range rather than individual breakpoint where values are consistent across the range.

Phase 6

Annotation reference

Claude generates two separate annotation references — one for Typography and one for Spacing — structured to match the Phase 5 comparison output. Each reference covers one section per frame and one breakpoint group per section, showing where discrepancies occur without replicating the full data from the comparison tables.

For module-level spacing (overall padding), use a simple pass/fail indicator per side rather than detailed annotations.

At the end of Phase 5, Claude will prompt: "Ready for me to generate the annotation references for Typography and Spacing? You can paste these into Figma once they're ready."

Annotation reference format

Each finding is output as a self-contained structured text block, formatted for direct paste into a Figma annotation frame. One block per finding. Blocks are separated by a divider line (---). The annotation references are plain text only — no markdown formatting, no HTML, no inline CSS — so they paste cleanly into Figma without conversion.

Each block opens with a status symbol and classification on the first line, followed by the section name and breakpoint, then consistent labelled fields:

  • ✕ FAIL — discrepancy, fix required. Include a Fix field.
  • ✓ DELTA — structural difference, visual result identical, no action needed. Include a Note field.
  • ⚠ DELTA — structural difference that may be fragile or worth a developer conversation. Include a Note field.

Block structure for a fail:

✕ FAIL — F[n]
[Section name] · [Breakpoint]
Property:  [property name]
Figma:     [Figma value]
Live:      [live value]
Fix:       [recommended fix]

Block structure for a delta:

✓ DELTA — D[n]
[Section name] · [Breakpoint]
Property:  [property name]
Figma:     [Figma value]
Live:      [live value]
Note:      [explanation — why this is considered equivalent]

Section names in annotation blocks must match the Phase 1 section map exactly. Do not use alternative labels or shorthand.

Verification

Spot-check a sample of extracted values manually in browser DevTools before finalising. Cross-reference findings against any prior audit documentation where it exists. Build developer confirmation into the handoff — when a fix is applied, the developer confirms the before value matched what was documented.

Reuse

The logical structure map, extraction script, and discrepancy table format are all reusable across modules. Only the element selectors and Figma values change. Each new module audit starts by updating the structure map for that component.

Known Challenges

🔍
DOM complexity on the live site Duplicate elements — such as mobile nav vs. hero panel link labels — require careful selector targeting to ensure the correct element is being measured and compared.
📐
Legacy breakpoint split at 1729px / 1730px A build artifact in the current codebase means desktop must be treated as two separate ranges: 992px–1729px and 1730px. Both must be extracted and compared independently — do not assume values are consistent across this boundary for any property category.
🖥️
Viewport width verification Set the browser window or DevTools responsive mode to the exact target breakpoint width. Chrome does not consume viewport width — always confirm with window.innerWidth before extracting values. If it does not match the target, investigate before proceeding.
